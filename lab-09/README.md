# BigQuery Lab 9 — Analyze Data with Gemini Assistance (OCBL436)

This lab introduces Gemini as an AI assistant within the Google Cloud and BigQuery environment. The business scenario is a proof of concept at Cymbal Superstore, where the goal is to evaluate whether Gemini can meaningfully support data analysts in generating, completing, and explaining SQL queries, and whether it can assist in building machine learning forecasting models.

The dataset used is `bigquery-public-data.thelook_ecommerce`, a public synthetic ecommerce and digital marketing dataset.

## Key Queries

**Task 4: Query explained by Gemini**

```sql
SELECT
    u.id AS user_id,
    u.first_name,
    u.last_name,
    AVG(oi.sale_price) AS avg_sale_price
FROM `bigquery-public-data.thelook_ecommerce.users` AS u
JOIN `bigquery-public-data.thelook_ecommerce.order_items` AS oi
    ON u.id = oi.user_id
GROUP BY 1, 2, 3
ORDER BY avg_sale_price DESC
LIMIT 10;
```

Gemini explained this as: a query that joins the users and order_items tables, groups by user, calculates each user's average sale price, and returns the top 10 highest-spending users.

**Task 5: SQL generated by Gemini from a natural language prompt**

Prompt used:
```
# Generate a BigQuery SQL query to calculate the daily total sales for each
# product. Join order_items with products using a LEFT JOIN on product ID.
# Include order date, product ID, product name, and total sales rounded to
# two decimal places. Group by column names, order by highest total sales.
```

Generated query:
```sql
SELECT
    DATE(order_items.created_at)        AS order_date,
    order_items.product_id,
    products.name                        AS product_name,
    ROUND(SUM(order_items.sale_price), 2) AS total_sales
FROM `bigquery-public-data.thelook_ecommerce.order_items` AS order_items
LEFT JOIN `bigquery-public-data.thelook_ecommerce.products` AS products
    ON order_items.product_id = products.id
GROUP BY
    order_date,
    order_items.product_id,
    product_name
ORDER BY total_sales DESC;
```

**Task 6: ARIMA_PLUS forecasting model**

```sql
CREATE MODEL bqml_tutorial.sales_forecasting_model
OPTIONS(
    MODEL_TYPE            = 'ARIMA_PLUS',
    time_series_timestamp_col = 'date_col',
    time_series_data_col  = 'total_sales',
    time_series_id_col    = 'product_id'
) AS
SELECT
    SUM(sale_price)     AS total_sales,
    DATE(created_at)    AS date_col,
    product_id
FROM `bigquery-public-data.thelook_ecommerce.order_items` AS t1
INNER JOIN `bigquery-public-data.thelook_ecommerce.products` AS t2
    ON t1.product_id = t2.id
GROUP BY 2, 3;
```

Forecast query generated by Gemini:
```sql
SELECT *
FROM ML.FORECAST(MODEL `bqml_tutorial.sales_forecasting_model`);
```

## Key Concepts

**Gemini in BigQuery**
Gemini integrates directly into the BigQuery query editor with three main features: auto-completion suggests SQL as you type, auto-generation creates full queries from natural language prompts, and explanation translates selected SQL into plain English. All three can be toggled on or off independently.

**Prompting for SQL generation**
Writing a good prompt for Gemini in BigQuery follows the same logic as writing a good query: be specific about the tables, the join type, the fields you want, and how you want results ordered. Vague prompts produce vague SQL. The more context you provide, the closer the output is to what you actually need.

**ARIMA_PLUS**
ARIMA_PLUS is BigQuery ML's built-in time series forecasting model type. It automatically handles seasonality, trend decomposition, and holiday effects. In this lab it was trained on historical daily sales per product and used to predict future sales periods. The model took approximately 10 minutes to train.

**ML.FORECAST**
The `ML.FORECAST` function queries a trained BigQuery ML time series model and returns predicted values along with confidence intervals for future periods. No additional configuration is needed beyond referencing the model name.

**Important note on Gemini output**
Gemini can generate output that sounds plausible but is factually incorrect. Always review and validate any SQL or explanation Gemini produces before using it in a production context. This applies especially to schema references and JOIN logic, where hallucinations are most likely to produce silent errors.
